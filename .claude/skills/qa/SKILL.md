対話型のQAセッションを実施する。ユーザーが遭遇した問題を聞き取り、コードベースを調査し、GitHub Issueを作成する。

## 各問題について

### 1. 聞き取りと軽い確認

ユーザーの言葉で問題を説明してもらう。確認の質問は**最大2〜3個**に絞る:

- 期待した動作 vs 実際の動作
- 再現手順（明らかでない場合）
- 常に起きるか、たまにか

過度に質問しない。説明が十分なら次に進む。

### 2. バックグラウンドでコードベースを調査

ユーザーと話しながら、Agent（subagent_type=Explore）をバックグラウンドで起動し、関連箇所を調査する。目的は修正方法を見つけることではなく:

- そのエリアで使われているドメイン用語を把握する
- 機能の本来の動作を理解する
- ユーザーから見える振る舞いの境界を特定する

この情報でより良い Issue を書く。ただし Issue 本文にはファイルパスや行番号を含めない。

### 3. スコープ判定: 単一 Issue か分割か

起票前に、**単一 Issue** か **複数に分割** すべきかを判断する。

**分割する場合:**

- 修正が複数の独立した箇所にまたがる
- 明確に分離可能で並行して作業できる
- 複数の異なる障害モードや症状がある

**単一にする場合:**

- 一つの場所で一つの振る舞いが間違っている
- 全ての症状が同じ原因から来ている

### 4. GitHub Issue を作成

`gh issue create` で作成する。ユーザーに確認を求めず、即座に起票してURLを共有する。

Issue は**リファクタリング後も有効**であること。ユーザーの視点で書く。

#### 単一 Issue のテンプレート

```
## 発生した問題

[ユーザーが経験した実際の動作を平易な言葉で記述]

## 期待する動作

[本来あるべき動作]

## 再現手順

1. [開発者が追える具体的な手順]
2. [コード内部のモジュール名ではなくドメイン用語を使う]
3. [関連する入力や設定を含める]

## 補足情報

[ユーザーの観察やコードベース調査で得た追加のコンテキスト — ドメイン用語を使い、ファイルは参照しない]
```

#### 分割する場合のテンプレート

依存順（ブロッカーを先に）で Issue を作成し、実際の Issue 番号で参照する。

各サブ Issue:

```
## 親 Issue

#<親Issue番号> または「QAセッションで報告」

## 何が問題か

[この特定の振る舞いの問題 — この部分だけを記述]

## 期待する動作

[この部分に対する期待される動作]

## 再現手順

1. [この Issue に固有の手順]

## ブロッカー

- #<Issue番号>（この Issue が別の Issue の修正を待つ必要がある場合）

または「なし — すぐに着手可能」

## 補足情報

[この部分に関連する追加の観察]
```

#### 分割時のルール

- **薄い Issue を多く** — 各 Issue が独立して修正・検証可能に
- **ブロック関係を正直に記載** — 本当に依存する場合のみ
- **依存順で作成** — 実際の Issue 番号を参照できるように
- **並列性を最大化** — 複数人（またはエージェント）が同時に取れるように

#### 全 Issue 共通ルール

- **ファイルパスや行番号は書かない** — すぐ古くなる
- **プロジェクトのドメイン用語を使う**
- **コードではなく振る舞いを記述する** — 「同期処理がパッチ適用に失敗する」であって「applyPatch() が42行目で throw する」ではない
- **再現手順は必須** — 特定できなければユーザーに聞く
- **簡潔に** — 開発者が30秒で読めること

起票後、全 Issue URL（ブロック関係のサマリ付き）を表示し「次の問題はありますか？」と聞く。

### 5. セッションの継続

ユーザーが終了と言うまで続ける。各 Issue は独立 — まとめない。
