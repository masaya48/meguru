ユーザーへのインタビュー、コードベース調査、モジュール設計を通じてPRDを作成し、GitHub Issueとして登録する。必要がなければステップをスキップしてよい。

## 手順

1. ユーザーに、解決したい問題と解決案の詳細な説明を求める。

2. リポジトリを調査し、ユーザーの主張を検証し、コードベースの現状を把握する。

3. 計画のあらゆる側面について、共通理解に到達するまで徹底的にインタビューする。設計ツリーの各分岐をたどり、決定間の依存関係を一つずつ解消していく。

4. 実装に必要なモジュールの概要を設計する。独立してテスト可能な「深いモジュール」を抽出する機会を積極的に探す。

深いモジュール（浅いモジュールの対義語）とは、多くの機能をシンプルでテスト可能な、滅多に変わらないインターフェースにカプセル化したものである。

ユーザーにモジュール構成が期待通りか確認する。テストを書くモジュールについても合意を取る。

5. 問題と解決策の完全な理解が得られたら、以下のテンプレートでPRDを作成し、`gh issue create` でGitHub Issueとして登録する。

## PRDテンプレート

```
## 課題

ユーザー視点で、解決すべき問題を記述する。

## 解決策

ユーザー視点で、解決策を記述する。

## ユーザーストーリー

全側面を網羅する、広範なユーザーストーリーのリスト:

1. <アクター>として、<機能>がほしい。なぜなら<得られる価値>だから

例: 住民として、回覧の既読状態が一目でわかる一覧画面がほしい。なぜなら未読の回覧を見逃さないようにしたいから

## 実装上の決定事項

合意した実装方針のリスト:

- 構築・修正するモジュール
- 変更するインターフェース
- 技術的な補足事項
- アーキテクチャ上の決定
- スキーマ変更
- APIコントラクト
- 特定のインタラクション

ファイルパスやコードスニペットは含めない（すぐに古くなるため）。

## テスト方針

テストに関する決定事項:

- 良いテストの基準（実装詳細ではなく外部の振る舞いをテストする）
- テスト対象のモジュール
- 参考になる既存テスト（コードベース内の類似テスト）

## スコープ外

このPRDのスコープに含めないもの。

## 補足

機能に関する追加の注記。
```
