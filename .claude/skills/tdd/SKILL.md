# テスト駆動開発

## 哲学

**基本原則**: テストは公開インターフェースを通じて振る舞いを検証する。実装の詳細ではない。コードは完全に変わっても、テストは壊れるべきではない。

**良いテスト**はインテグレーション的: 公開APIを通じて実際のコードパスを実行する。システムが「何をするか」を記述し、「どうやるか」は問わない。良いテストは仕様書のように読める — 「有効なカートでチェックアウトできる」を見ればどんな機能か分かる。内部構造に依存しないので、リファクタリングに耐える。

**悪いテスト**は実装に結合している。内部のコラボレータをモックし、プライベートメソッドをテストし、インターフェースではなく外部手段（DBに直接クエリ等）で検証する。警告サイン: リファクタリングでテストが壊れるが、振る舞いは変わっていない。

## アンチパターン: 水平スライス

**全テストを先に書いて、その後に全実装を書いてはいけない。**

これは「水平スライス」— RED を「全テストを書く」、GREEN を「全コードを書く」と扱うこと。

正しいアプローチ: **垂直スライス（トレーサーバレット）**。1テスト → 1実装 → 繰り返し。

```
NG（水平）:
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

OK（垂直）:
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
```

## ワークフロー

### 1. 計画

コードを書く前に:

- [ ] ユーザーにインターフェースの変更を確認
- [ ] テストすべき振る舞いを確認（優先順位付き）
- [ ] 深いモジュールの機会を探す（小さいインターフェース、深い実装）
- [ ] テスト可能なインターフェースを設計
- [ ] テストする振る舞いをリストアップ（実装ステップではない）
- [ ] ユーザーの承認を得る

**全てをテストすることはできない。** どの振る舞いが最も重要かユーザーに確認する。

### 2. トレーサーバレット

システムについて1つのことを確認する1つのテストを書く:

```
RED:   最初の振る舞いのテストを書く → テスト失敗
GREEN: 通すための最小限のコードを書く → テスト成功
```

### 3. インクリメンタルループ

残りの各振る舞いについて:

```
RED:   次のテストを書く → 失敗
GREEN: 通すための最小限のコード → 成功
```

ルール:

- 一度に1テスト
- 現在のテストを通すのに十分なコードだけ
- 将来のテストを先読みしない
- 観察可能な振る舞いに集中

### 4. リファクタリング

全テストが通った後、リファクタリング候補を探す:

- [ ] 重複の抽出
- [ ] モジュールを深くする（複雑さをシンプルなインターフェースの裏に隠す）
- [ ] SOLID 原則を自然に適用
- [ ] 新しいコードが既存コードの問題を明らかにしていないか
- [ ] 各リファクタリング後にテストを実行

**RED の状態でリファクタリングしない。** 先に GREEN にする。

## サイクルごとのチェックリスト

```
[ ] テストは実装ではなく振る舞いを記述している
[ ] テストは公開インターフェースのみ使用
[ ] テストは内部リファクタリングに耐える
[ ] コードはこのテストに対して最小限
[ ] 投機的な機能を追加していない
```

## 良いテストと悪いテスト

### 良いテスト

```typescript
// GOOD: 観察可能な振る舞いをテスト
test("有効なカートでチェックアウトできる", async () => {
  const cart = createCart();
  cart.add(product);
  const result = await checkout(cart, paymentMethod);
  expect(result.status).toBe("confirmed");
});

// GOOD: インターフェースを通じて検証
test("ユーザー作成後に取得できる", async () => {
  const user = await createUser({ name: "Alice" });
  const retrieved = await getUser(user.id);
  expect(retrieved.name).toBe("Alice");
});
```

### 悪いテスト

```typescript
// BAD: 実装詳細をテスト
test("checkout calls paymentService.process", async () => {
  const mockPayment = jest.mock(paymentService);
  await checkout(cart, payment);
  expect(mockPayment.process).toHaveBeenCalledWith(cart.total);
});

// BAD: インターフェースを迂回して検証
test("createUser saves to database", async () => {
  await createUser({ name: "Alice" });
  const row = await db.query("SELECT * FROM users WHERE name = ?", ["Alice"]);
  expect(row).toBeDefined();
});
```

## モックのルール

**システム境界でのみモックする:**

- 外部API（LINE, 決済等）
- データベース（テストDBが使えない場合のみ）
- 時刻 / 乱数
- ファイルシステム（場合による）

**モックしない:**

- 自分のクラス / モジュール
- 内部のコラボレータ
- 自分がコントロールするもの

## 深いモジュール

「A Philosophy of Software Design」より:

**深いモジュール** = 小さいインターフェース + 多くの実装

インターフェース設計時に問う:

- メソッド数を減らせるか？
- パラメータをシンプルにできるか？
- より多くの複雑さを内部に隠せるか？

## テスト可能なインターフェース設計

1. **依存を受け取る、作らない**（DI）
2. **副作用ではなく結果を返す**
3. **小さい表面積**（メソッドが少ない = テストが少なくて済む）
